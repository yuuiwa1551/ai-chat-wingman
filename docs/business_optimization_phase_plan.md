# AI Chat Wingman Business Optimization Phase Plan

本计划补充 `docs/ai_chat_wingman_spec_plan.md`，聚焦“业务可用性”和“用户首次成功体验”。实现时仍按 `AGENTS.md` 约束执行：每次只闭环一个 P，不一次性做完所有方向。

## P0 - First Successful Reply Loop

目标：新用户在不导入、不配置真实 Provider 的情况下，也能理解产品边界，并快速看到一条可复制的候选回复。

范围：

1. 初次导入页提供“一键试用示例聊天”，使用默认人设进入主工作台。
2. 主工作台明确标识 Mock 演示模式，说明它只用于跑通流程，不代表真实回复质量。
3. 回复生成区提供场景目标快捷项，而不是只暴露抽象参数。
4. 候选回复展示“为什么这样写”的简短解释，帮助用户判断是否可用。

验收：

1. 未配置 Provider 时，用户仍能进入主界面并基于示例聊天理解核心流程。
2. 用户能通过场景按钮快速切换回复目标、语气、主动程度和风险边界。
3. 候选卡片能说明每条回复的策略差异。
4. 前端构建通过。

## P1 - Provider Setup Confidence

目标：降低真实模型配置门槛，让用户知道 mock、OpenAI-compatible provider、模型检测和连通测试之间的关系。

范围：

1. Provider 设置页增加分步提示：选择类型、填写 Base URL、填写 API Key、检测模型、测试连通。
2. 对 mock 模式、真实 provider 缺失、模型检测失败、连通失败给出不同文案。
3. 保存 provider 后在主界面给出清晰状态，例如“真实模型已启用”或“仍在演示模式”。

验收：

1. 新用户不看 README 也能完成 OpenAI-compatible provider 配置。
2. mock 模式和真实 provider 的预期差异明确。
3. 检测模型与测试连通的错误反馈能指向下一步。

## P2 - Target Value Front Loading

目标：让“对象档案”不只是资料库，而是让用户立刻看到它会怎样影响回复。

范围：

1. 对象侧栏展示该对象影响回复的 2-3 条规则。
2. 导入或新建对象后生成可读的回复策略摘要。
3. 回复区在选择对象时提示当前对象策略如何影响候选。

验收：

1. 用户选中对象后能看懂“为什么这次回复会更谨慎/更主动/更短”。
2. 对象资料空时仍有明确的手动补充入口。
3. 不把长期记忆直接写入 approved，仍遵守 pending 确认机制。

## P3 - Memory Confirmation Simplification

目标：降低长期记忆维护成本，让用户只确认真正有价值的记忆。

范围：

1. 记忆建议分成“建议保存”“不确定”“不建议保存”三类。
2. 支持批量确认、忽略和拒绝。
3. 每条建议解释来源和影响范围。

验收：

1. 用户能批量处理低风险记忆。
2. 任何长期记忆进入 approved 前都经过确认。
3. 拒绝或忽略不会影响后续生成链路。

## P4 - Public Positioning And Release Readiness

目标：把项目对外表达从“功能列表”转成“本地、可控、不自动发送的聊天回复助手”。

范围：

1. README 增加适合人群、核心边界、快速试用路径和当前限制。
2. Release 文案与 README 保持一致，强调 Windows preview、mock 演示和本地数据边界。
3. 后续公开截图优先展示主工作台、初次导入页和 Provider 设置页。

验收：

1. 新访客能在 1 分钟内理解项目能做什么、不能做什么。
2. 用户不会误解为自动读取聊天软件或自动发送消息。
3. README 与实际 UI 流程一致。
