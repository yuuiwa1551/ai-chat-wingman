# v0.1.4 后续实现计划（风格分析 jobs 化 + 密钥本地加密）

本轮在 v0.1.3 的基础上继续，目标是把最后一个可能超时的 LLM 任务异步化，并提升本地密钥的静态安全性。

## P0-A 风格校准分析走 jobs 表

### 现状
- `POST /style-test/sessions/{id}/analysis` 直接 `await analyze_style_test_session`，可能阻塞 HTTP 请求超过 2 秒。
- 截图解析、对象整理已经走 jobs，本任务是最后一个同步 LLM 入口。

### 方案
- 后端：新增 `run_style_analysis_job(job_id, payload)`，懒加载 `analyze_style_test_session`，写入 `{analysis, profile, llm_call_id}`。
- 端点：`analyze_session` 改为同步 `def`，先校验 session 存在（404），再 `create_job(job_type="style_analysis", payload={session_id})` + `BackgroundTasks.add_task`，返回 `{job_id, status}`。
- 前端：`analyzeStyleTest` 改为提交 + `pollJobResult`，复用现有轮询能力；`StyleTestPanel` 增加取消/进度。

### 验收
- `POST .../analysis` 返回 `job_id`，轮询 `/jobs/{id}` 拿到 `analysis/profile/llm_call_id`。
- 现有相关测试更新为轮询模式后通过。

## P0-B Provider 密钥本地静态加密

### 现状
- `llm.providers` 以明文 JSON 存在 `app_settings`，`api_key` 直接落库。
- 任何人拿到 SQLite 文件即可读取明文 key。

### 方案（零新增第三方依赖）
- 新增 `app/security/secret_box.py`：本地密钥文件（数据目录下 `secret.key`，0600 权限）+ HMAC-SHA256 keystream 的可逆加密，密文带版本前缀 `enc:v1:`。
- `settings_store` 写入 provider 时对 `api_key` 加密、读取时解密；对非加密历史值（明文）保持兼容并在下次写入时升级为密文。
- 仅加密 `api_key` 字段，不动其它 provider 配置；mask 行为不变。
- purge `include_settings` 时一并清理密钥文件。

### 验收
- 新建/更新 provider 后，SQLite 中 `api_key` 为 `enc:v1:` 开头密文，不含明文。
- 读取/路由/测试 provider 仍能拿到明文 key 正常调用。
- 旧明文配置可被正确解密读取并在写入时升级。

## 执行顺序
P0-A → P0-B → 验证（pytest + build）→ 升版本 0.1.4 + release notes + README → 提交并打 tag 自动发版。
